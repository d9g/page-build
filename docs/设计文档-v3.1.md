> **更新日期：** 2026-04-12
> **产品定位：** 极简 AI 智能排版工具
> **技术路线：** 微信原生小程序 + FastAPI 后端

## 一、项目概述

### 目标

做一个极简的公众号文章排版工具。用户粘贴/输入文字 → 一键 AI 排版 → 微调 → 复制到公众号。

### 核心理念

**最简单的方式实现最实用的功能。** 首页只有一个输入框和一个「AI 智能排版」按钮。

### 技术选型

| 层级 | 技术 | 理由 |
|------|------|------|
| **小程序前端** | 微信原生小程序（WXML + WXSS + JS） | 广告无障碍、微信能力直接调用 |
| **后端** | Python FastAPI | 异步支持好，与 AI 模型对接方便 |
| **AI 大模型** | **智谱 GLM-4-Flash**（永久免费） | 128K 上下文，免费无限调用 |
| **数据库** | SQLite（MVP）→ MySQL（后续） | MVP 阶段轻量够用 |
| **缓存** | Redis | 验证码、会话、频率限制 |
| **部署** | Nginx + uvicorn + systemd | 反向代理 + 进程管理 |

### AI 大模型选型详情

| 参数 | 值 | 说明 |
|------|-----|------|
| **模型** | `glm-4-flash` | 智谱 AI 永久免费模型 |
| **API 地址** | `https://open.bigmodel.cn/api/paas/v4/chat/completions` | 官方 API |
| **上下文窗口** | 128K tokens | 输入 + 输出总计 |
| **最大输出** | 4096 tokens（默认），最高可设 131072 | 排版输出一般 4096 够用 |
| **费用** | ¥0（永久免费） | 核心优势 |
| **速率限制** | 有并发限制（免费版基础额度） | 需注意高峰期排队 |

### 为什么选原生小程序而不是 web-view 套壳？

| 维度 | 套壳 H5 | 原生小程序 | 结论 |
|------|---------|-----------|------|
| 广告集成 | ❌ web-view 与广告组件不共存 | ✅ 随意放置 Banner/插屏/激励视频 | 原生胜 |
| 微信能力 | ⚠️ 需要 JS Bridge 桥接 | ✅ 直接调用 wx.login / 分享 / 订阅消息 | 原生胜 |
| 更新速度 | ✅ 即时生效 | ⚠️ 需审核 1-3 天 | 套壳胜 |
| H5 复用 | ✅ 一套代码两端用 | ❌ 需另做 H5 版 | 套壳胜 |

> **结论：** 对于本项目的核心目标（广告变现 + 公众号导流），原生方案的优势是决定性的。更新速度的劣势可以通过「前端薄、后端厚」的架构设计来化解——把可变逻辑全部放后端下发，日常迭代 90% 无需提审。

#### rich-text 预览渲染方案

> IMPORTANT: 小程序的 rich-text 组件渲染 HTML 有两种模式：直接传 HTML 字符串，或传 nodes 数组。对于需要动态调整样式的场景，使用 nodes 数组更灵活。

```javascript
// 将后端返回的排版结果转为 rich-text nodes
function buildNodes(sections, theme, adjustments) {
  const nodes = []

  for (const section of sections) {
    switch (section.type) {
      case 'title':
        nodes.push({
          name: 'h1',
          attrs: {
            style: `color:${theme.titleColor};font-size:${adjustments.fontSize + 5}px;font-weight:bold;text-align:center;margin:20px 0 16px;`
          },
          children: [{ type: 'text', text: section.content }]
        })
        break

      case 'paragraph':
        nodes.push({
          name: 'p',
          attrs: {
            style: `color:${theme.bodyColor};font-size:${adjustments.fontSize}px;line-height:${adjustments.lineHeight};margin-bottom:${adjustments.paragraphGap}px;text-indent:2em;`
          },
          children: [{ type: 'text', text: section.content }]
        })
        break

      case 'quote':
        nodes.push({
          name: 'div',
          attrs: {
            style: `background:${theme.quoteColor};border-left:3px solid ${theme.quoteBorderColor};padding:12px 16px;margin:12px 0;border-radius:4px;`
          },
          children: [{
            name: 'p',
            attrs: { style: `color:#666;font-size:${adjustments.fontSize - 1}px;margin:0;` },
            children: [{ type: 'text', text: section.content }]
          }]
        })
        break

      case 'divider':
        nodes.push({
          name: 'div',
          attrs: {
            style: 'text-align:center;color:#ccc;margin:20px 0;letter-spacing:8px;'
          },
          children: [{ type: 'text', text: theme.dividerStyle === 'dots' ? '· · ·' : '———' }]
        })
        break
    }
  }

  return nodes
}
```

#### slider 防抖优化

```javascript
// 避免拖动 slider 时频繁重建 nodes 导致卡顿
let debounceTimer = null

function onSliderChange(field, value) {
  // 立即更新数字显示
  this.setData({ [`adjustments.${field}`]: value })

  // 防抖 200ms 后才重新渲染预览
  clearTimeout(debounceTimer)
  debounceTimer = setTimeout(() => {
    const nodes = buildNodes(this.data.sections, this.data.currentTheme, this.data.adjustments)
    this.setData({ previewNodes: nodes })
  }, 200)
}
```

## 四、关注验证机制（单账号聚焦推广 + 验证码方案）

### 核心策略

> IMPORTANT: **每次只推一个公众号，集中火力涨粉。** 运营人工指定当前推哪个号，切换时机选在凌晨低峰期（如 2:00-4:00），避免出现「用户正在关注 A 号，突然切成 B 号导致验证码对不上」的情况。

```
运营策略示意：

第 1-2 周：推「老杨说事」 → 集中涨粉 500
        ↓ （凌晨 2 点切换）
第 3-4 周：推「老杨生活」 → 集中涨粉 500
        ↓ （凌晨 2 点切换）
第 5-6 周：推「老杨日签」 → 集中涨粉 500
        ↓
轮回 / 根据运营数据动态决定
```

公众号池里的**每个公众号都需要配置**消息回调（因为不知道什么时候会切到该号）：

2. **自动回复规则**：收到关键词「排版」→ 后端生成验证码 → 回复

```python

async def wechat_callback(account_id: str, request: Request):

        if active_id and account_id != active_id:
            # 用户关注的不是当前推广的号，但也给验证码
            # 验证时再校验是否匹配当前激活账号
            # NOTE: 这样做的好处是——即使切换了推广账号，
            # 之前关注过池内其他号的用户发「排版」也能拿到验证码
            pass

        code = str(random.randint(1000, 9999))

        await redis.setex(
            f"verify_code:{code}",
            300,  # 5 分钟过期
            json.dumps({
                "account_id": account_id,

        )
```

### 小程序侧验证

```python
@router.post("/api/v1/verify")
async def verify_follow(request: VerifyRequest):
    """
    校验验证码
    
    验证逻辑：
    1. 验证码必须有效（Redis 中存在且未过期）
    2. 验证码对应的公众号必须在池内（任意一个均可）
    3. 验证通过后记录状态，30 天免重复验证
    
    NOTE: 不强制要求验证码来自「当前激活账号」，
    只要是池内任一公众号产生的验证码即可通过。
    这样做是为了避免切换推广账号时，刚好在操作中的用户验证失败。
    """
    code_data = await redis.get(f"verify_code:{request.code}")
    if not code_data:
        return {"success": False, "message": "验证码无效或已过期"}

    info = json.loads(code_data)

    # 检查 account_id 是否在公众号池内
    pool_ids = [acc["id"] for acc in ACCOUNT_POOL]
    if info["account_id"] not in pool_ids:
        return {"success": False, "message": "验证码无效"}

    # 记录验证状态到数据库
    await db.save_verification(
        mini_openid=request.openid,
        account_id=info["account_id"],
        expires_at=datetime.now() + timedelta(days=30)
    )

    await redis.delete(f"verify_code:{request.code}")

    return {"success": True, "valid_days": 30}
```

### 运营管理接口（切换推广账号）

> IMPORTANT: 切换推广账号的操作应在**凌晨使用频率最低时**（建议 2:00-4:00）进行，避免正在操作的用户受到影响。

```python
        "previous": old_active,
        "current": request.account_id,
        "message": f"已切换到 {request.account_id}"
```

### 切换安全机制

为什么切换要在凌晨低峰期？看这个时序问题：

```
❌ 高峰期切换的风险：

16:00 用户小明打开小程序 → 弹窗显示「请关注老杨说事」
16:01 小明长按二维码 → 跳转到公众号「老杨说事」→ 关注
16:02 运营此时切换推广账号 → 改为「老杨生活」  ⬅ 危险！
16:03 小明在「老杨说事」发送「排版」→ 获得验证码 7823
16:04 小明回到小程序输入 7823 → 验证通过 ✅（因为池内任一号均可通过）
      但如果弹窗已刷新为「老杨生活」，用户会困惑

✅ 凌晨切换就没有这个问题：

03:00 运营切换推广账号 → 改为「老杨生活」
      此时几乎无人使用，无时序冲突
08:00 新用户打开小程序 → 弹窗正常显示「老杨生活」
```

> NOTE: 虽然验证逻辑设计为「池内任一公众号的验证码均可通过」，
> 不会导致功能故障，但用户看到弹窗中途变了会感到困惑。
> 所以切换时机选在凌晨是体验层面的考虑。

## 五、AI 排版核心逻辑

### 模型参数与 Token 预算分析

> IMPORTANT: 输入字数上限不能拍脑袋定，需要根据 GLM-4-Flash 的实际参数来计算。

**GLM-4-Flash 参数：**

| 参数 | 值 |
|------|-----|
| 上下文窗口 | 128K tokens |
| 最大输出 | 4096 tokens（排版场景建议值） |
| 中文 token 换算 | 约 1 个汉字 ≈ 1.5-2 个 tokens |

**Token 预算分配：**

```
128K 总预算（实际安全使用 100K 以内）

┌─────────────────────────────────────────┐
│  System Prompt（排版指令）   ≈ 800 tokens │
│  用户输入文本               ≈ ? tokens  │
│  AI 输出（排版 JSON 结果）   ≈ 输入的 1.5-2倍│
│  安全余量                   ≈ 10%      │
└─────────────────────────────────────────┘

假设用户输入 8000 字中文：
  输入 tokens ≈ 8000 × 1.5 = 12000 tokens
  输出 tokens ≈ 12000 × 2  = 24000 tokens（排版后 JSON 比原文长）
  System Prompt              = 800 tokens
  总计 ≈ 36800 tokens ← 远在 128K 安全范围内 ✅

假设用户输入 20000 字中文：
  输入 tokens ≈ 20000 × 1.5 = 30000 tokens
  输出 tokens ≈ 30000 × 2   = 60000 tokens
  总计 ≈ 91000 tokens ← 接近上限，风险大 ⚠️
  而且 max_tokens 需设为 60000+，响应时间可能 30s+ ❌
```

**结论：输入上限定为 8000 字**

| 因素 | 8000 字 | 20000 字 |
|------|---------|----------|
| Token 安全 | ✅ 充裕（约 37K / 128K） | ⚠️ 接近上限 |
| 响应时间 | ✅ 3-8 秒 | ❌ 30 秒+ |
| 输出质量 | ✅ 稳定 | ⚠️ 长文容易丢失结构 |
| 用户体验 | ✅ 等待可接受 | ❌ 等太久会放弃 |
| 公众号文章长度 | ✅ 大多数文章 < 5000 字 | 超长文章极少 |

> NOTE: 8000 字覆盖了绝大多数公众号文章场景。真正超过 8000 字的长文，后续可以考虑「分段排版」方案（将文章按章节拆分，逐段调用 AI）。

### AI 调用代码

```python
import httpx
import os

# 智谱 AI 配置
ZHIPU_API_URL = "https://open.bigmodel.cn/api/paas/v4/chat/completions"
ZHIPU_API_KEY = os.getenv("ZHIPU_API_KEY")  # NOTE: 从环境变量读取，不要硬编码

async def call_glm4_flash(
    system_prompt: str,
    user_content: str,
    max_tokens: int = 4096,
    temperature: float = 0.3
) -> str:
    """
    调用智谱 GLM-4-Flash 模型
    temperature 设为 0.3，排版场景需要较低的随机性
    """
    async with httpx.AsyncClient(timeout=60.0) as client:
        resp = await client.post(
            ZHIPU_API_URL,
            headers={
                "Content-Type": "application/json",
                "Authorization": f"Bearer {ZHIPU_API_KEY}"
            },
            json={
                "model": "glm-4-flash",
                "messages": [
                    {"role": "system", "content": system_prompt},
                    {"role": "user", "content": user_content}
                ],
                "temperature": temperature,
                "max_tokens": max_tokens
            }
        )
        result = resp.json()
        return result["choices"][0]["message"]["content"]
```

### Prompt 版本管理系统

#### 设计原则

```yaml
  6. 保持原文含义不变，不添加、不删减、不改写内容

  - 只做排版，不改变原文内容和含义
  - 不要添加原文中没有的内容
```

#### Prompt 管理器代码

```python
# services/prompt_manager.py
import yaml
import os
import random
import logging
from pathlib import Path
from functools import lru_cache

logger = logging.getLogger(__name__)

PROMPTS_DIR = Path(__file__).parent.parent / "prompts"


class PromptManager:
    """
    Prompt 版本管理器
    负责加载、缓存和切换 Prompt 版本
    修改 YAML 文件后自动生效，无需重启服务
    """

    def __init__(self):
        self._cache = {}
        self._config_mtime = 0

    def _load_config(self) -> dict:
        """加载全局配置，检测文件变化时自动刷新"""
        config_path = PROMPTS_DIR / "config.yaml"
        mtime = os.path.getmtime(config_path)

        if mtime != self._config_mtime:
            with open(config_path, "r", encoding="utf-8") as f:
                self._config = yaml.safe_load(f)
            self._config_mtime = mtime
            logger.info(f"Prompt 配置已刷新，激活版本: {self._config['active_version']}")

        return self._config

    def _load_version(self, version: str) -> dict:
        """加载指定版本的 Prompt 文件"""
        if version not in self._cache:
            path = PROMPTS_DIR / f"{version}.yaml"
            with open(path, "r", encoding="utf-8") as f:
                self._cache[version] = yaml.safe_load(f)
            logger.info(f"Prompt 版本 {version} 已加载")

        return self._cache[version]

    def get_active_prompt(self) -> tuple[dict, str]:
        """
        获取当前应使用的 Prompt
        支持 A/B 测试时按权重随机分配版本

        返回: (prompt_data, version_id)
        """
        config = self._load_config()

        # A/B 测试模式
        if config.get("ab_test", {}).get("enabled"):
            versions = config["ab_test"]["versions"]
            weights = [v["weight"] for v in versions]
            chosen = random.choices(versions, weights=weights, k=1)[0]
            version = chosen["version"]
        else:
            version = config["active_version"]

        prompt_data = self._load_version(version)
        return prompt_data, version

    def reload(self):
        """强制清除缓存，重新加载所有文件"""
        self._cache.clear()
        self._config_mtime = 0
        logger.info("Prompt 缓存已清除")


# 全局单例
prompt_manager = PromptManager()
```

### 微信公众号 HTML 兼容性适配

> WARNING: 微信公众号编辑器对 HTML/CSS 支持非常受限，AI 生成的 HTML 必须经过清洗适配。

**微信公众号编辑器的关键限制：**

| 支持 | 不支持 |
|------|--------|
| 内联 `style` 属性 | `class`、`id`、外联 CSS |
| `div`、`p`、`span`、`h1-h6` | `position`、`float` |
| `blockquote`、`ul`、`ol`、`li` | Flexbox、Grid |
| `img`（需微信域名素材链接） | `@font-face`、自定义字体 |
| `table`、`tr`、`td` | JavaScript |
| `a`（仅公众号内链接） | 外部链接、iframe |
| `color`、`font-size`、`margin`、`padding` | `animation`、`transition` |
| `background-color`、`border` | `box-shadow`（部分不生效）|

**后端 HTML 清洗逻辑：**

```python
import re
from bs4 import BeautifulSoup

# 微信公众号编辑器不支持的 CSS 属性，需要自动移除
UNSUPPORTED_CSS_PROPERTIES = [
    "position", "float", "display", "flex", "grid",
    "animation", "transition", "transform",
    "box-shadow", "text-shadow",
    "overflow", "z-index"
]

def sanitize_html_for_wechat(html: str) -> str:
    """
    清洗 HTML，确保 100% 兼容微信公众号编辑器
    """
    soup = BeautifulSoup(html, "html.parser")

    for tag in soup.find_all(True):
        # 移除 class 和 id 属性
        tag.attrs = {
            k: v for k, v in tag.attrs.items()
            if k == "style" or k == "src" or k == "href"
        }

        # 清洗 style 中不支持的属性
        if tag.get("style"):
            style = tag["style"]
            for prop in UNSUPPORTED_CSS_PROPERTIES:
                style = re.sub(rf"{prop}\s*:[^;]+;?", "", style)
            tag["style"] = style.strip()

    return str(soup)
```

### 小程序端 SSE 接收

```javascript
// 小程序中使用 wx.request 的 enableChunked 模式接收 SSE
function requestLayout(content, options) {
  return new Promise((resolve, reject) => {
    const sections = []

    const task = wx.request({
      url: 'https://pb.d9g.com.cn/api/v1/layout',
      method: 'POST',
      enableChunked: true,
      header: {
        'Authorization': `Bearer ${getToken()}`,
        'Content-Type': 'application/json'
      },
      data: { content, options },
      success: () => resolve(sections),
      fail: reject
    })

    // 监听分块数据
    task.onChunkReceived((res) => {
      const text = arrayBufferToString(res.data)
      const events = parseSSE(text)

      for (const event of events) {
        if (event.event === 'section') {
          sections.push(event.data)
          // 通知页面追加渲染新的 section
          getApp().globalData.onSectionReceived?.(event.data)
        } else if (event.event === 'complete') {
          // 排版完成，保存完整结果
          getApp().globalData.layoutResult = event.data
        } else if (event.event === 'progress') {
          // 更新加载提示文案
          getApp().globalData.onProgress?.(event.data.message)
        }
      }
    })
  })
}
```

### 频率限制实现

```python
from fastapi import HTTPException
import time

async def check_rate_limit(openid: str, action: str = "layout"):
    """
    滑动窗口频率限制
    排版接口：每用户每小时最多 10 次
    """
    key = f"rate_limit:{action}:{openid}"
    now = time.time()
    window = 3600  # 1 小时窗口

    pipe = redis.pipeline()
    pipe.zremrangebyscore(key, 0, now - window)  # 清理过期记录
    pipe.zadd(key, {str(now): now})              # 添加当前请求
    pipe.zcard(key)                               # 计算窗口内请求数
    pipe.expire(key, window)                      # 设置 key 过期
    results = await pipe.execute()

    count = results[2]
    if count > 10:
        raise HTTPException(
            status_code=429,
            detail="排版次数已达上限，请 1 小时后再试"
        )
```

### 封装 request 工具

```javascript
// utils/request.js
const BASE_URL = 'https://pb.d9g.com.cn'

/**
 * 封装 wx.request
 * 自动携带 token、统一错误处理、超时提示
 * @param {string} path API 路径
 * @param {object} options 请求配置
 */
function request(path, options = {}) {
  return new Promise((resolve, reject) => {
    const token = getApp().globalData.token

    wx.request({
      url: `${BASE_URL}${path}`,
      method: options.method || 'GET',
      data: options.data,
      header: {
        'Content-Type': 'application/json',
        'Authorization': token ? `Bearer ${token}` : '',
        ...options.header
      },
      timeout: options.timeout || 30000,
      enableChunked: options.enableChunked || false,
      success(res) {
        if (res.statusCode === 401) {
          // token 过期，重新登录
          getApp().reLogin().then(() => {
            request(path, options).then(resolve).catch(reject)
          })
          return
        }
        if (res.statusCode === 429) {
          wx.showToast({ title: '操作太频繁，请稍后再试', icon: 'none' })
          reject(new Error('RATE_LIMITED'))
          return
        }
        if (res.statusCode >= 400) {
          wx.showToast({
            title: res.data?.detail || '请求失败',
            icon: 'none'
          })
          reject(new Error(res.data?.detail || 'REQUEST_FAILED'))
          return
        }
        resolve(res.data)
      },
      fail(err) {
        wx.showToast({ title: '网络连接失败', icon: 'none' })
        reject(err)
      }
    })
  })
}

module.exports = { request, BASE_URL }
```

## 十一、错误处理与边界场景

| 场景 | UI 表现 | 技术处理 |
|------|---------|---------|
| **AI 接口超时（>30s）** | 加载动画消失，弹窗「排版超时，是否重试？」 | request timeout + 重试按钮 |
| **AI 返回质量差** | 结果页提供「🔄 重新排版」按钮 | 调 `/layout/retry` 接口，使用不同 temperature |
| **网络断开** | 全屏提示页「网络似乎不太好」+ 重试按钮 | `wx.getNetworkType` 监听 |
| **输入内容过长** | 输入框下方红色提示 + 禁止继续输入 | 前端 maxlength + 后端校验 |
| **输入内容过短** | 排版按钮置灰 + 提示「至少 50 字」 | 前端校验 |
| **复制失败** | 弹窗「复制失败，请手动长按选择复制」 | `wx.setClipboardData` 的 fail 回调 |
| **内容违规** | 弹窗「内容包含敏感信息，请修改后重试」 | 微信 msgSecCheck 返回结果 |
| **频率限制** | Toast「操作太频繁，请 1 小时后再试」 | 429 状态码 |
| **验证码过期** | 输入框下方红字「验证码已过期，请重新获取」 | 后端返回过期状态 |
| **小程序更新** | 启动时检测新版本，提示「发现新版本，需要重启」 | `wx.getUpdateManager` |

### 草稿自动保存

```javascript
// pages/index/index.js

// 输入内容变化时自动保存草稿
onInput(e) {
  const content = e.detail.value
  this.setData({
    content,
    wordCount: content.length
  })

  // 防抖保存：内容变化后 3 秒无操作自动保存
  clearTimeout(this.saveDraftTimer)
  this.saveDraftTimer = setTimeout(() => {
    wx.setStorageSync('draft_content', content)
    wx.setStorageSync('draft_time', Date.now())
  }, 3000)
},

// 页面加载时检测草稿
onLoad() {
  const draft = wx.getStorageSync('draft_content')
  const draftTime = wx.getStorageSync('draft_time')

  if (draft && draft.length > 0) {
    // 计算草稿距今多久
    const minutes = Math.floor((Date.now() - draftTime) / 60000)
    const timeText = minutes < 60
      ? `${minutes} 分钟前`
      : `${Math.floor(minutes / 60)} 小时前`

    wx.showModal({
      title: '发现未完成的草稿',
      content: `${timeText}保存的草稿（${draft.length}字），是否恢复？`,
      confirmText: '恢复',
      cancelText: '忽略',
      success: (res) => {
        if (res.confirm) {
          this.setData({ content: draft, wordCount: draft.length })
        } else {
          wx.removeStorageSync('draft_content')
          wx.removeStorageSync('draft_time')
        }
      }
    })
  }
}
```

#### 数据库设计

```sql
-- 用户发布权益表
CREATE TABLE publish_plans (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,
    plan_type TEXT NOT NULL,           -- 'per_use' / 'monthly' / 'yearly'
    remaining_count INTEGER DEFAULT 0, -- 剩余发布次数（按次付费）
    expires_at TIMESTAMP,              -- 卡到期时间（月卡/年卡）
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id)
);

-- 用户授权的公众号表
CREATE TABLE user_gzh_auth (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,
    authorizer_appid TEXT NOT NULL,     -- 授权方 appid
    authorizer_access_token TEXT,       -- 授权方 access_token
    authorizer_refresh_token TEXT,      -- 用于续期
    account_name TEXT,                  -- 公众号名称
    token_expires_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id)
);

-- 发布日志表
CREATE TABLE publish_logs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,
    layout_id INTEGER,                 -- 关联的排版记录
    title TEXT,
    media_id TEXT,                      -- 微信草稿 media_id
    images_uploaded INTEGER DEFAULT 0,
    images_failed INTEGER DEFAULT 0,
    status TEXT DEFAULT 'success',      -- 'success' / 'failed'
    error_message TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

#### 微信 API 依赖

| API | 用途 | 备注 |
|-----|------|------|
| `cgi-bin/token` | 获取 access_token | 用户授权的公众号 token |
| `cgi-bin/media/uploadimg` | 上传正文图片 | 返回微信 CDN URL |
| `cgi-bin/material/add_material` | 上传封面图 | 返回 thumb_media_id |
| `cgi-bin/draft/add` | 推送草稿箱 | 返回 media_id |

> NOTE: 上述 API 需要微信开放平台「第三方平台」资质，或用户自己的公众号 AppID/AppSecret。
> 简化方案：用户在小程序中填写自己的 AppID + AppSecret，
> 后端直接调用其公众号 API。需用户将服务器 IP 加入微信 IP 白名单。

#### 安全注意事项

| 风隩 | 措施 |
|------|------|
| 用户 AppSecret 存储 | AES 加密存储，不以明文保存 |
| access_token 泄露 | token 只在后端使用，不传到前端 |
| 超额发布 | 后端严格校验发布次数，不信前端 |
| IP 白名单 | 引导用户添加服务器 IP，提供教程链接 |
